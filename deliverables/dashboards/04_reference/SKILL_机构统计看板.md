# Skill: 发行台账簿记明细录入

**版本**: v2.0
**上一稳定版**: v1.0（归档于 `skills/簿记录入/v1.0_stable/`）
**更新日期**: 2026-05-16

## 适用场景
- **全量录入**：用户提供1个发行台账Excel + N个项目的簿记明细Excel，要求将各个项目的簿记申购数据录入到发行台账对应的项目区域中
- **增量合并**（v2.0新增）：用户提供上周已加工台账（含WXY）+ 本周新原始台账 + 本周簿记明细，自动识别增量项目并合并，避免对存量项目重复劳动

## 触发关键词
- 全量录入：录入簿记数据、录入到发行台账、簿记明细录入、补充簿记数据
- 增量合并：增量台账合并、本周新项目、增量录入、合并新台账、台账增量

## 工作流程

1. **用户输入**：发行台账文件 + 多个项目的簿记明细文件

2. **预处理：取消合并单元格 + 向下填充**
   - 检测发行台账Excel中的所有合并单元格
   - 全部取消合并（unmerge）
   - 对因取消合并产生的空单元格以及表中其他空白单元格，从上一行向下填充，补全所有A~V列数据
   - ⚠ 向下填充时，**必须继承源单元格格式**（字体、字号、对齐方式、边框、数字格式等），确保填充单元格与源行格式一致（这是硬约束，见"约束与边界"第6条）
   - 确保每一行数据独立完整（为后续按行匹配操作打下基础）
   - ⚠ 注意：P列（分层）、U列（认购机构）和V列（认购份额）**不参与向下填充**，保持原样。P列也是合并单元格，只在分层首行有值，如果参与填充会导致次级/中间级/受益权的分层信息全部丢失（详见踩坑#29）
   - 预处理完成后，**必须立即验证**：随机抽查3~5行P/U/V列值是否与原始台账一致

3. **重新确认项目区域**：预处理（取消合并+向下填充）可能导致行数变化，必须在此步骤重新定位每个项目的起始/结束行
   - 通过A列项目名定位每个项目的行区域范围
   - 记录每个项目的（start_row, end_row），供后续录入使用
   - ⚠ 此步骤不可省略：预处理前的行号可能已失效

4. **执行录入**：调用 insert_bookkeeping.py 脚本，多项目一次性处理
   - 读取预处理后的台账文件
   - 使用步骤3确认的行区域范围
   - 为每个项目读取对应的簿记明细
   - 在内存中完成匹配、合并、排序、填充
   - 一次性写入最终结果

5. **输出**：台账文件名后加 "-补充簿记数据v1.xlsx"，如有迭代则递增版本号

6. **数据与格式统一优化（自检查前执行）**：⚠ **此步骤覆盖全表所有数据行（row 3 到 last_data_row），不是只处理WXY项目行。预处理填充的行、原始数据行、新增行全部纳入格式优化范围。** 在所有录入完成后执行以下统一格式优化：
   - **L列（簿记时间）**：值为 `datetime` 类型的单元格，统一设为 `yyyy-mm-dd` 短日期格式
   - **M列（国股CD）、T列（成本）、Q列（分层占比）**：数值型单元格统一设为 `0.00%` 百分比格式
   - **N列（ALL-IN成本）**：数值型单元格统一设为 `0.000%` 百分比格式（保留原始3位小数精度）
   - **R列（对应金额）**：数值型单元格统一设为 `0.00_);[Red]\(0.00\)` 格式（保留负数红色括号）
   - **V列（认购份额）**：⚠ 不参与格式优化，保留原始格式不做任何修改（V列属于数据保护范围）
   - **全表表头（行1~2）**：华文楷体/11号/加粗/居中/细边框（颜色 indexed=64）
   - ⚠ **全表数据行（含预处理填充行+原始数据行+新增行）**：华文楷体/11号/居中/四边细边框（原始台账中KaiTi字体行保留不覆盖）。不能只处理WXY相关行，必须扫描全表所有有数据的行
   - **W列（申购利率）**：数值→`0.00%`，"自持"等字符串保持原样
   - **Y列（申购规模）**：数值→`0.00` 两位小数

7. **自检查（输出前执行）**：在所有录入和格式优化步骤完成后，输出文件给用户之前，脚本自动执行以下检查：

   ### 7.1 簿记数据完整性检查
   - **明细条目数量检查**：对比每个项目明细Excel的原始申购记录数 vs 台账中录入的W列数据行数（考虑同利率合并导致的减少，告知用户"原始N笔，合并后M笔，差额原因"）
   - **申购规模合计检查**：明细Excel中申购规模合计 vs 台账Y列申购规模合计，两边金额必须一致（精确到小数点后2位）
   - **交叉验证逻辑**：将台账WXY列按项目汇总，与明细Excel的原始汇总逐项比对

   ### 7.2 原始数据保护检查
   - **U列（认购机构）检查**：抽取台账原始U列值与处理后U列值对比，确认未被覆盖或修改
   - **V列（认购份额）检查**：同上，确认未被修改
   - **其他原始列抽样检查**：检查A~T列表中若干关键字段未被篡改

   ### 7.3 格式与公式检查
   - **错误值检查**：扫描全表，检查是否存在 `#N/A`、`#VALUE!`、`#REF!`、`#DIV/0!`、`#NAME?` 等错误值
   - **公式引用检查**：检查是否有不应存在公式的列（如WXY手工录入列）意外包含公式
   - **格式检查（全表抽样）**：⚠ 不能只检查WXY项目行，必须对**全表数据行**抽样检查格式一致性
     - 抽样规则：每50行抽1行 + 所有WXY项目行
     - 检查维度：数字格式（W列`0.00%`/Y列`0.00`）+ 边框（四边thin）+ 字体（华文楷体或KaiTi/11号）+ 对齐（居中）
     - 任一抽样行格式不通过，判定为格式检查失败

   ### 7.4 分级问题处理

   #### 阻断级问题（禁止输出文件，等待用户明确指令）
   - 明细条目数量不一致且无法用合并规则解释
   - 申购规模合计不一致（差异 > 0.01）
   - 原始数据（A~V列）被篡改或覆盖
   - 表中出现错误值（#N/A、#VALUE!、#REF! 等）

   #### 警告级问题（列出详情，询问用户是否继续输出）
   - 数字格式偏差（如某单元格百分比格式漏设）
   - 不应存在公式的列中发现了公式（非WXY列）
   - WXY列格式不完整（缺少标题或数字格式）

   #### 通用规则
   - 任何检查项不通过时，必须如实汇报差异详情（差异数值、差异行号、差异内容）
   - **不做任何预设假设**（如"我判断这个差异是合理的所以忽略"）
   - 阻断级问题：禁止输出文件，等待用户明确指令
   - 警告级问题：列出问题后询问用户是否继续输出

## 调用方式

```python
# 多项目模式（推荐）
python3 insert_bookkeeping.py 台账.xlsx --multi \
  "项目名1:明细1.xlsx" \
  "项目名2:明细2.xlsx" \
  --output 台账-补充簿记数据v1.xlsx

# 单项目模式
python3 insert_bookkeeping.py 台账.xlsx 明细.xlsx 项目名 --output 台账-补充簿记数据v1.xlsx

# 继续录入（基于已有输出追加新项目）
  # ⚠ 注意：增量追加可能导致行偏移错误，下游项目数据可能被上游扩展的WXY行侵占。
  # 优先推荐从预处理原始文件一次性处理所有项目，而非分步追加。必须增量追加时，
  # 执行后务必检查上游项目的WXY数据是否侵占了相邻项目区域。
  python3 insert_bookkeeping.py 台账-补充簿记数据v1.xlsx --multi \
  "新项目:新明细.xlsx" \
  --output 台账-补充簿记数据v2.xlsx
```

## 核心逻辑（脚本自动完成）

### 数据保护
- 原表A~V列（U列认购机构、V列认购份额）**绝对不修改、不覆盖**
- 新增数据只写入W列（申购利率）、X列（穿透机构）、Y列（申购规模）

### 合并规则
- **同机构同利率同分层合并规模**（多笔同机构同利率且同分层的申购合并为一条，规模求和）
- ⚠ **跨分层禁止合并**：同一机构在不同分层（如优先A和优先B）的申购即使利率相同也不可合并，必须分行展示
- **不同利率分行展示**（单行单元格不拼接多个利率）
- **同机构多条记录连续排列**（不同利率行排在一起）

### 排列顺序
- 优先A行全部在前，优先B行全部在后，不交叉
- 同机构其余新增行紧跟原行之后
- 台账中不存在的机构追加到所属分层（优先A/优先B）的末尾，同机构连续排列
- 若分属不同分层，按明细所在sheet/区域确定分层归属，同机构在不同分层分别追加

### 填充规则
- A~T列向下填充（新增行的空白单元格从上一行继承），⚠ 但**P列（分层）跳过不填充**
- P列（分层）、U列（认购机构）和V列（认购份额）跳过不填充，保持空值
- ⚠ 填充时**必须继承源单元格格式**（Font/Alignment/Border/NumberFormat），不仅是值（这是硬约束，详见踩坑#12/#31）

### 格式
- W列（申购利率）：`0.00%` 百分比格式；字符串"自持"保持原样
- X列（穿透机构）：文本；**优先使用台账简写**（如台账写"中信证券投行"，X列不写"中信证券（投行上报）"；台账写"光大证券投行"，X列不写"光大证券（投行）"），保持表格信息简洁一致
- Y列（申购规模）：`0.00` 两位小数格式
- **WXY列标题**：必须设置表头文字（申购利率 / 穿透机构 / 申购规模），字体/加粗/对齐由最后一步格式统一优化统一处理
- **WXY列数据行格式**：由最后一步格式统一优化统一处理
- **向下填充时**：填充单元格的字体、字号、对齐方式、边框线必须继承自源单元格
- ⚠ **格式优化范围=全表所有有数据的行**：不能只处理"被修改的行"或"WXY项目行"，必须扫描row 3到last_data_row的全部数据行

### 数据和格式统一优化（自检查之前，脚本自动执行）
在写入WXY列数据并完成基础格式设置后，执行全表格式统一优化：
- **L列（簿记时间）**：`yyyy-mm-dd` 短日期格式
- **M列（国股CD）**：百分比格式 `0.00%`
- **N列（allin成本）**：百分比格式 `0.000%`（保留原始3位小数精度）
- **Q列（分层占比）、T列（成本）**：百分比格式 `0.00%`
- **R列（对应金额）**：两位小数格式 `0.00_);[Red]\(0.00\)`（保留负数红色括号）
- **V列（认购份额）**：⚠ 不修改格式，保留原始格式不做任何操作（V列属于数据保护范围）
- **全表表头（行1~2）**：华文楷体/11号/粗体/居中/细边框
- **全表数据行**：华文楷体/11号/居中/细边框（原始台账中KaiTi字体行保留不覆盖）

## 项目名映射规则

明细文件名（如"京诚15号8期_簿记.xlsx"）与台账E列项目名（如"京诚15-8"）可能不一致，需要映射：

1. **精确匹配**：文件名去除"_簿记.xlsx"后缀后与E列完全一致
2. **去序号修饰词**：去掉"号""第"等修饰词，如"15号8期"→"15-8"，"21第5期"→"21-5"
3. **去括号/书名号**：去掉"《》""「」"等包裹符号
4. ⚠ **匹配失败时必须立即报告**：如果任何一份明细文件无法匹配到台账项目，脚本应终止并报告，不能让项目静默跳过
5. **匹配确认步骤**：脚本启动时先列出所有明细文件→台账项目的匹配结果，确认无遗漏后再执行录入

## 机构名自动映射规则

1. 先精确匹配（台账名 = 明细名）
2. 硬编码映射（优先于模糊匹配）：
   - **杭州联合银行→杭州农商行（资管）** — 杭州联合银行资管与杭州农商行资管为同一机构
   - **杭州联合→杭州联合银行** — 明细使用简称"杭州联合"，台账用全称
   - **交银理财多通道→交银理财** — 同一机构的不同渠道写法
   - **中信证券投行→中信证券（投行上报）** — 简称写法差异
   - **工银瑞信→工银瑞信基金** — 明细漏掉"基金"后缀
3. 包含关系匹配："中信证券"包含在"中信证券（投行上报）"中
4. 去后缀匹配：去掉"基金""资管""理财""投行""有限公司""有限责任公司"等后缀后尝试包含
5. **全称截断匹配**：当明细机构为法定全称（如"中信证券股份有限公司（投行上报）""杭州联合农村商业银行股份有限公司（资产管理部）"）时，去掉"股份有限公司""有限责任公司"等公司后缀词，仅保留括号前的前两段核心名称，再与台账简称匹配。
   - 例："中信证券股份有限公司（投行上报）" → 截断为"中信证券" → 匹配台账"中信证券投行"
   - 例："杭州联合农村商业银行股份有限公司（资产管理部）" → 截断为"杭州联合农村商业银行" → 进一步去后缀"农村商业银行"→"杭州联合" → 匹配台账"杭州联合银行"
6. **通道/渠道名映射**：同一机构的"通道""渠道"变体写法（如"交银理财多通道"→"交银理财"）在hard_map中统一映射，不依赖自动去后缀。

## 架构说明（多项目处理）
- **核心架构**：先将所有项目的原始行数据读入内存，然后在内存中构建新行布局，最后一次性写入工作表
- **从前往后逐段拼接**：按原始行号升序逐项目处理，中间行以原始行号索引读取，项目区域以原始`end_row+1`跳过，不做偏移补偿
- **一次性清空写入**：所有计算完成后，清空旧数据并写入最终结果（避免中间保存丢失数据）

## 约束与边界

1. **不可篡改原始数据**：台账中已有的任何单元格值和格式（A~V列、表头等）均不可修改或覆盖，包括但不限于：单元格数值、数字格式、字体、对齐、边框。如需修改原始数据值（如修正认购机构、调整认购份额），必须先询问用户确认，获得明确许可后方可操作。特别强调：V列（认购份额）的数字格式也属于保护范围，不做任何格式统一优化。

2. **T/U/V列零篡改原则**：T列（成本）、U列（认购机构）、V列（认购份额）是台账原始数据，向下填充时必须跳过这些列。V列的数字格式也属于保护范围，不做任何格式统一优化。无论自动还是手动处理，输出后必须与原始版本对比验证T/U/V列完全一致。

3. ⭐ **P列（分层）也是合并单元格，向下填充时必须跳过**：P列只在分层首行有值（如"优先级""中间级""次级""受益权"），其余行合并显示。如果P列参与向下填充，会导致所有行P列变成"优先级"，次级/中间级的分层信息丢失，进而导致匹配错乱（次级U行被优先A明细匹配）。详见踩坑#29/#35。

4. **忠实保留明细全量信息**：簿记明细表中的每一行申购数据都必须被使用——要么匹配到台账机构中录入WXY列，要么作为未匹配明细追加到对应分层末尾。未匹配机构的处理规则：追加到所属分层末尾、同机构连续、X列标注原始机构名。若分属不同分层，按明细所在sheet/区域确定分层归属。不可遗漏、不可丢弃、不可选择性忽略任何一条申购记录。处理完成后，应确保每一条明细数据都有唯一落脚位置。

5. **追加行按分层归位**：未匹配的明细行必须追加到对应分层（优先A追加行跟在优先A段后，优先B跟在优先B段后），不能统一堆在末尾。

6. **U/V保护：同一原始U行仅匹配一次**：台账中同机构同分层只有一行原始U/V数据。明细中同机构有多笔申购时，只有第一条能被匹配写入该原始行；其余明细作为未匹配追加（WXY有数据，U/V为空）。不可将同一原始U/V复制到多行WXY数据上。⚠ 匹配时必须限定在同分层范围内，不同分层的U行不可跨层匹配（详见踩坑#35）。

7. ⭐ **格式一致性硬约束**：输出文件中，全表所有数据行（包括预处理填充行、原始数据行、新增行）的格式必须统一。具体要求：
   - 字体：华文楷体/11号（原始台账中KaiTi字体行保留不覆盖）
   - 对齐：水平居中+垂直居中
   - 边框：四边细边框（thin）
   - 格式优化范围=**全表所有有数据的行**（row 3到last_data_row），不能只处理"被修改的行"或"WXY项目行"
   - 预处理向下填充时，必须继承源单元格的Font/Alignment/Border/NumberFormat，不能只复制值（详见踩坑#12/#31/#32）
   - QC检查必须全表抽样（每50行抽1行 + 所有WXY项目行），不能只检查WXY范围

## 特殊格式处理：多sheet簿记明细

某些项目的簿记明细以多sheet格式存在（如"尚赢2-4"，优先A/优先B分属不同sheet），脚本的`read_detail_file()`不支持此类格式。

### 处理流程

1. **优先尝试转化为标准格式**：如果明细数据列顺序与标准3列一致（利率、机构、规模），可提前将多sheet合并为单文件单sheet的标准格式，然后走脚本流程。
2. **必须手动处理时**：
   - 从原始预处理文件的原始区域读取尚赢2-4的A~V列
   - 分别读取明细各sheet并合并到内存
   - **匹配规则**：一个原始U/V行只能匹配一次明细；同机构多笔申购时，只有第一条匹配原始U行，其余作为未匹配追加
   - **按分层放置**：优先A匹配行不变 → 优先A未匹配追加行 → 优先B匹配行不变 → 优先B未匹配追加行 → 中间级/受益权保持原始
   - **向下填充范围仅限于A~T列**，U/V列跳过不填充
   - **填充只在同分层内进行**，不同分层之间不跨越
3. **输出后立刻验证**：用对比脚本对比v2原始版和v3手动版的T/U/V列完全一致。

### 特殊情况：多sheet + 宽表（列结构不同）

当明细同时满足以下两个条件时，属于"多sheet+宽表"组合场景，不能通过简单列选择后合并走脚本：

- 优先A/优先B分属不同sheet（多sheet）
- 列数超过标准3列，且列顺序不同（如包含证券名称、认购金额、边际利率等额外字段）

**处理策略**：
1. 先确认每个sheet中**利率、机构、规模**分别在哪一列（标准3列的列号可能跨sheet一致）
2. 分别读取各sheet，只提取这3列数据
3. 将两个sheet的数据合并为一个列表（优先A在前、优先B在后）
4. 用合并后的列表走手动匹配流程（同"必须手动处理时"的匹配/分层/填充规则）
5. 不得尝试将宽表整体作为标准3列传递给脚本——脚本会识别错误并崩溃或静默误写入

### 验证脚本
```python
# 对比原始版本和手动版本中同一项目的T/U/V列
v2: T=0.0167(优先A)/0.0183(优先B)/0.031(中间级)/0.0515(受益权)
v3: 应与v2完全一致，不得改变
# 手动逐行对比
for r in range(start_row, end_row+1):
    t2, u2, v2 = ws2.cell(r,20), ws2.cell(r,21), ws2.cell(r,22).value
    t3, u3, v3 = ws3.cell(r,20), ws3.cell(r,21), ws3.cell(r,22).value
    assert (t2 == t3) and (u2 == u3) and (v2 == v3), f"Row {r}: T/U/V不匹配"
```

## 增量台账合并（v2.0新增）

### 适用场景
用户每周收到一份新的发行台账（含存量+增量项目），已在上一周加工后的台账中录入了簿记明细（WXY列）。需要自动化增量合并：识别新项目，从新台账提取增量数据，追加到加工后台账，仅对新项目执行簿记录入。

### 前置假设
- 老项目的A~V列在新台账中不变
- 项目名（E列）在不同周次始终一致

### 工作流程

1. **用户输入**：上周已加工台账（含WXY） + 本周新原始台账（无WXY） + 本周簿记明细文件（0~N个）

2. **预处理两个台账**
   - 对**已加工台账A**：取消合并+向下填充，⚠ **W/X/Y列（23/24/25）与P/U/V一起单独保护**——WXY不是合并单元格，但向下填充会把W列值扩散到空行，必须保护
   - 对**新原始台账B**：正常预处理（取消合并+向下填充，保护P/U/V）
   - 预处理完成后，各记录项目行范围

3. **项目差分——识别增量**
   - 提取A的所有项目名集合 `set_A`
   - 提取B的所有项目名集合 `set_B`
   - 增量项目 = `set_B - set_A`
   - ⚠ 输出增量项目清单，**必须用户确认无遗漏后再继续**
   - 如果 `set_A - set_B` 非空（老项目在新台账中消失），⚠ 警告用户

4. **从B中提取增量项目的完整A~V行**
   - 按E列定位每个增量项目在B中的行范围
   - 提取该范围内所有行的 A~V列数据（含P列分层、U/V原始值）
   - 记录每个增量项目的 `(项目名, B中的start_row, B中的end_row)`

5. **将增量项目追加到A末尾**
   - 在A的最后一个数据行之后，逐行追加增量项目的A~V数据
   - 每个增量项目的行结构保持与B中一致（包括P列分层、U/V原始数据）
   - 追加行的格式统一设置：华文楷体/11号/居中/细边框

6. **对增量项目执行簿记录入**
   - 复用现有 `insert_bookkeeping.py` 的匹配/合并/排序/写入逻辑
   - ⚠ **只处理增量项目，存量项目的WXY数据保持原样不动**
   - 无明细文件的增量项目，保留原始A~V数据不动，WXY为空

7. **全表格式优化 + QC**
   - 对增量项目新增的行执行格式优化（同步骤6格式规则）
   - 对全表数据行执行格式统一优化（确保格式一致性）
   - **QC重点**：
     - 7.1 增量项目簿记数据完整性（同全量录入QC）
     - ⭐ 7.2 **存量项目WXY数据保护**：对比A和D中所有存量项目的WXY列，确认完全一致，任一差异即判定为阻断级问题
     - 7.3 全表格式一致性抽样

### 调用方式

```python
# 增量合并模式（v2.0新增）
python increment_merge.py \
  --processed 上周已加工台账.xlsx \
  --new-raw 本周新原始台账.xlsx \
  --details 明细1.xlsx 明细2.xlsx ... \
  --output 本周已加工台账.xlsx

# 全量录入模式（v1.0兼容，不变）
python insert_bookkeeping.py 台账.xlsx 明细1.xlsx [...] output.xlsx
```

### 增量合并特有约束

1. ⭐ **已加工台账预处理的WXY保护**：A~V列向下填充时，W/X/Y列（23/24/25）必须与P/U/V一起单独保护，否则W列值会被填充扩散到空行
2. ⭐ **存量WXY数据不可篡改**：增量合并后，A中所有存量项目的W/X/Y列值必须与输入完全一致，不允许任何差异
3. **增量项目追加到末尾**：新项目追加在A的最后数据行之后，不插入到中间
4. **增量项目确认步骤**：识别增量项目后，必须输出清单让用户确认，不能静默执行
