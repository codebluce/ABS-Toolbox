ABS工具箱

一套从台账录入、机构统计到发行定价分析的端到端 ABS 业务工具箱。把散落在三套脚本里的簿记、统计、定价能力，整合成一条可串行调用的自动化工作流。

端到端工作流 三大核心模块 A→B→C 审计闭环 纯本地 Excel 处理

产品概览

一个入口，三大模块，一条流水线。台账进，看板出。

ABS工具箱把原本分散的三套独立 skill（发行定价、机构统计、簿记录入）整合进单一代码源，共享同一套底座与机构名映射，避免重复维护。业务同事只需提供一份台账 Excel，即可依次得到补充簿记后的台账、机构统计看板、以及三张发行定价分析看板。

模块一 · 簿记录入

把当日簿记明细，稳妥地并入历史台账 —— 不错行、不重复、不覆盖。

簿记录入模块解决的是「增量并表」这一最容易出错的环节。业务同事每天拿到的是零散的当日簿记明细，而台账是一份持续累加的定稿文件。手工复制粘贴极易出现错位、重复项、覆盖旧记录。increment_merge.py 用项目主键匹配 + 结构化对齐，自动完成这项合并。

两种工作模式

补充簿记模式

已有台账 + 补录簿记信息

台账里项目已存在，只是簿记字段（利率、规模、簿记日等）尚未回填。按项目主键定位到对应行，只补写缺失字段，不新增行。

增量合并模式

已有台账 + 新增项目

当日簿记里出现台账中没有的新项目，追加为新行并保持列结构一致，同时对已存在项目执行补充簿记逻辑。

手工并表 vs 工具并表

手工复制粘贴
  • 凭肉眼对齐行，易错位
  • 同一项目可能被重复追加
  • 补录时误覆盖已有历史值
  • 合并单元格结构被打乱
increment_merge
  • 项目主键精确匹配定位行
  • 去重逻辑防重复追加
  • 只补空字段，历史值保护
  • 保留台账原有列/结构

调用示例

bash 
set PYTHONUTF8=1
python scripts/increment_merge.py ^
  --ledger  deliverables/ledger/03_final/2026年ABS发行台账-定稿.xlsx ^
  --booking deliverables/ledger/05_booking/当日簿记明细-0705.xlsx ^
  --mode    increment

v2.5.0 更新：匹配规则调优，rebook（重簿记回填）成为默认行为，减少历史项目二次录入时的人工干预。

模块二 · 机构统计

一份台账，三个视角。按管理人、托管行、销售机构，自动汇总规模与项目数。

gen_institution_stats.py 从台账出发，围绕三类合作机构维度做聚合统计，产出一张交互式看板 HTML。看板支持按维度切换，展示各机构的项目数、发行规模、占比排名，帮助业务快速回答「今年哪家管理人做得最多」「销售集中度如何」这类问题。

机构名归一：内置机构名映射表，自动合并同一机构的不同简称/全称（如申万宏源相关主体合并统计），避免因命名不一致导致的统计分裂。

视角一 · 管理人统计

按计划管理人聚合，展示各管理人承做的 ABS 项目数与发行规模，识别头部管理人。

模块三 · 发行定价

三个定价工具，回答三个问题：成本贵不贵、机构比不比得过、利差合不合理。

发行定价是工具箱里工具最密集的模块，由三个 gen 脚本组成，分别从不同角度剖析 ABS 的发行成本，共产出三张分析看板。

工具一 · 成本分布 gen_abs_cost_report.py

统计发行成本的分布形态，识别异常高/低成本项目（outliers），辅助判断某笔发行的定价是否偏离市场区间。

工具三 · 利差分析 gen_spread_report.py

计算发行利率相对基准的利差，跟踪利差随时间/评级/期限的变化，评估定价的合理性。

调用示例

bash 
set PYTHONUTF8=1
python scripts/gen_abs_cost_report.py --ledger 台账.xlsx --out 成本看板.html
python scripts/gen_compare_tool.py    --ledger 台账.xlsx --out 比对看板.html
python scripts/gen_spread_report.py   --ledger 台账.xlsx --out 利差看板.html

案例：成本分布工具曾定位到一批异常成本记录（见 cost_outliers_correct.csv），复核后确认为数据录入口径问题，避免了将其纳入市场基准的误判。

全流程编排

三步串行：录入 → 统计 → 定价。一条命令链，从原始簿记到全套看板。

三大模块并非孤立运行，它们构成一条自然的流水线。点击每一步可展开细节。

1
录入 · 簿记合并
当日簿记明细并入定稿台账
运行 increment_merge.py，得到「补充簿记后的台账.xlsx」。这是后续所有统计与定价的唯一数据源，确保口径一致。
2
统计 · 机构汇总
基于合并后台账生成机构统计看板
运行 gen_institution_stats.py，产出管理人/托管行/销售机构三视角看板 HTML，归档至 dashboards/。
3
定价 · 成本分析
生成成本/比对/利差三张定价看板
依次运行三个 gen 脚本，产出发行定价分析看板。至此，一份台账已转化为完整的一套业务看板。

配套设施：归档工具 abs_archive.py 统一管理 dashboards / ledger / index；五层自检脚本 self_check.py 提供 auto / full / degraded 三档降级校验，跑流程前先体检。

审计子系统

A 送审 → B 审计 → C 归档。每一次工具或规则变更，都走一遍三角色闭环。

工具箱内建一套轻量审计流程，用于管控脚本与匹配规则的变更质量。它以文件为载体，在 audit/ 目录下按 submissions / reviews / closed 三个阶段流转。

A · 送审

变更方整理改动说明与自检结果，提交到 submissions，等待审计。

B · 审计

审计方复核变更，出具 verdict（通过 / 有条件通过 / 打回），写入 reviews。

C · 归档

留档遗留条件，把闭环记录移入 closed，更新索引。

为什么需要它

无审计
  • 匹配规则改动无人复核
  • 变更历史散落，难追溯
  • 遗留问题无处留档
A→B→C 闭环
  • 每次变更强制复核
  • submissions/reviews/closed 全程留痕
  • verdict + 遗留条件结构化归档

写在最后

台账进，看板出。中间那些容易错的手工活，交给工具箱。

ABS工具箱把簿记、统计、定价三件事收进一条流水线，用机构名归一保证口径一致，用 A→B→C 审计守住变更质量。它不改变业务逻辑，只是把重复、易错、难追溯的环节自动化、留痕化。
