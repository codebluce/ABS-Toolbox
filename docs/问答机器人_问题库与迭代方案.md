## 三、缺口清单与迭代方案（P0 → P2）

### P0 —— 会给出**错误的确定数字**或空结果，损害信任，先修

**P0-1 年月组合冲突（B5）**
- 根因：`parseTime()` 的月份区间用 `DATA_YEAR`(2026) 拼年，与 `filters.year` 相与恒空。
- 方案：`parse()` 先解析年份，把 `yearsExplicit` 传入 `parseTime(q, explicitYear)`；单年份时月份区间用该年拼；多年份+月份 → 时间条件退位，按"年份分组+各年同月"处理（groupBy=year，dateFrom/To 不设，改在 `filterRecs` 里按 `r.date` 的月份分量过滤）。
- 验收：`2025年3月交银理财投了多少` → 非空、芯片显示 年份2025+时间3月。

**P0-2 "去年/前年"裸用无时间条件（B6）**
- 根因：`parseTime()` 只有月份/今年等模式返回区间，"去年"单独出现时返回 null。
- 方案：统一走年份筛选——`parseYears()` 识别 去年→[DATA_YEAR-1]、前年→[DATA_YEAR-2]（比拼日期区间更契合多年份架构）。
- 验收：`去年一共投了多少` → 年份芯片 2025年，数字=2,720.28亿（2025年全量）。

**P0-3 追问继承失真（H1）**
- 根因：继承判定 `hasFilter = Object.keys(filters).length > 0`，年份命中即视为"有筛选"而放弃继承。
- 方案：拆分 `hasEntityFilter`（排除 year 后是否有实体条件）；"只有年份变化 + 追问线索词"时继承上一问实体，仅替换年份。
- 验收：`交银理财白条投了多少` → `那2024年呢` → 芯片含 交银理财+白条系+2024年+承接标记。

**P0-4 指标抢占（G1 的一半）**
- 根因：`parseMetric()` 顺序 cost 在 count 前，"成本高于2%的有多少笔"被判成 cost。
- 方案：count/proj 这类**计数问法**（多少笔/几笔/多少个项目）优先级提到最前——计数意图明确度远高于名词性的"成本"。
- 验收：该问句指标芯片=记录数（阈值部分见 P1-5）。

### P1 —— 常见问法答不上/答不全

**P1-1 裸年份范围（B2，用户点名的场景）**
- 方案：`parseYears()` 的 range 正则去掉对"年"字的强制：`/20(2[4-6])\s*[-~－到至]\s*20(2[4-6])/`（两个 20XX 夹连接符已足够无歧义）；**单个**裸年份仍要求"年"字（防项目编号误伤）。
- 验收：`交银理财2024-2026白条优先A档分别投资多少亿` → 与 B1 完全等价的三行年度对比。

**P1-2 多实体"分别/各自/谁多"→ 按该字段分组（C2/C3）**
- 方案：`parse()` 里检测：某实体字段命中值 ≥2 且句含 /分别|各自|谁(投|买|认购)?得?(多|少)|对比/ → `groupBy=该字段`；"谁投得多"额外置 topN=命中值数量。复用现有分组表格渲染。
- 验收：C2 → 两行（交银理财 X亿 / 中邮理财 Y亿）；C3 → 同表且第一行即答案。

**P1-3 极值方向（D4）**
- 方案：识别 /最低|最少|最小/ → `spec.extremeDir='asc'`；`groupAgg` 排序尊重方向；"哪家/哪个/谁"+指标 → 自动 groupBy 对应字段 + topN=3（给前3便于核对，首行加粗为答案）。
- 验收：`成本最低的机构是哪家` → 按平均成本升序前3的机构表。

**P1-4 占比（E1）**
- 方案：识别 /占.{0,6}(比例|比重|百分之|多少)/ → 计算分子（当前全部条件）与分母（去掉"主语实体字段"保留时间/年份的全量），回答给三个数：分子、分母、百分比，芯片注明分母口径。
- 验收：E1 → "119.35亿 / 1,611.56亿 = 7.4%"。

**P1-5 成本/利差阈值（G1 的另一半）**
- 方案：仿照现有份额阈值，识别 /(成本|利率|利差)(高于|低于|超过|大于|小于)\s*([\d.]+)\s*%/ → `costMin/costMax/spreadMin/spreadMax`，`filterRecs` 补过滤。
- 验收：`成本高于2%的有多少笔` → 计数，芯片含"成本>2%"。

**P1-6 多资产/多分层子串 OR（C4）**
- 方案：子串兜底从"每字段仅取一个最长 span"改为**循环消费**：命中后把 span 标记 consumed，继续找下一个最长未消费 span，直到无新命中；每轮命中值并入同字段 OR。
- 验收：`白条和金条一共投了多少` → 资产芯片同时含白条系与金条系。

### P2 —— 体验增强与工程化

| # | 缺口 | 方案要点 |
|---|---|---|
| P2-1 | 近N年（F2） | `parseTime` 增加 /近\|最近\s*([一二两三\d])\s*年/ → 从最新日期回溯 N 年 |
| P2-2 | 历年/逐年/各年（F3） | 触发 groupBy=year（全部年份），趋势词+"年"粒度时年度聚合优先于逐月 |
| P2-3 | 机构类别概念（I3） | 新增可选分类词表 `INST_CATEGORY = {理财子: n=>/理财$/.test(n), 券商自营: ...}`，命中类别→展开为机构名 OR 集合并在芯片展示"理财子(28家)"；同时给 layer 子串兜底加防误伤：兜底命中值若含括号备注类长尾值需整词边界 |
| P2-4 | 同义词表重构 | 内联 `.replace('夹层',...)` 链抽成 `SYNONYM` 常量表（词→标准值），业务同事可直接提词表 PR，不碰逻辑 |
| P2-5 | 问题库固化 | 本文件第二节全部问题写入 `scripts/lab/itl_chat_questions.txt`，harness 缺省跑全量 |

**实施顺序建议**：P0 一批（4项都是小改动、互不耦合，一次提交+一轮回归）→ P1 分两批（1/2/3 一批，4/5/6 一批）→ P2 按需。每批改动跑 harness 全量回归 + 浏览器抽查 2 题 + 双推。

## 四、最佳实践（沉淀为本模块迭代规范）

1. **测试先行的问答回归自检**：新增能力先把问题加进题库、跑出"改前基线"，再改代码，改后 diff——与仓库既有"5层自检"同构，这是问答版。
2. **宁可"没听懂"，不硬答错数**：金融工具里一个自信的错误数字比报错危险得多（本轮 H1/B5/G1 三处实测都是这种类型）。任何解析不确定的分支，回退到"没听懂+例句"，不做静默猜测。
3. **条件回显 = 可核对性契约**：任何新增的筛选维度/口径（占比分母、阈值、极值方向）都必须出现在回答的条件芯片里，用户能一眼核对机器的理解。
4. **词表零维护原则**：实体词表永远从内嵌数据自动生成；人工维护的只有两张小表——同义词表（口语→标准值）与类别表（概念→机构集合规则），集中声明、与逻辑解耦。
5. **面板联动一致性**：chat 给出的每个数字都必须能在下方面板复现（同条件筛选后卡片数字一致），这是每轮回归的抽查项。
6. **多 Agent 协作纪律**：改 `itl_chat.js` 前先跑 harness 留存基线输出（附在送审报告），改后 diff 两份输出——审计人无需读懂全部代码即可确认无回退。
