## 一、模块架构速览

`gen_integrated_dashboard.main()` 采用"编排层 + 数据驱动"设计，流程清晰：

1. 各子模块 `compute_data(xlsx_path)` → 返回数据 dict（内含 QC precheck）
2. 各子模块 `render_body(data)` → 返回 HTML 片段
3. 拼接 4 份 CSS + `TAB_CSS`，套 Tab 框架，写文件 + 轻量 QC

架构优点：
- 已摆脱旧版"依赖中间 HTML + BeautifulSoup"的脆弱做法，改为数据驱动，值得肯定。
- `compute_data` / `render_body` 计算与渲染分离，职责清晰。
- 末尾有 panel 数量与 JS 函数存在性的结构自检，属良好实践。

## 二、P0 — 同一 Excel 被重复预处理约 9 次（首要瓶颈）

### 现象
核心重操作 `abs_common.preprocess_xlsx_for_pandas()` 用 openpyxl **完整加载工作簿 → 遍历拆除全部合并单元格 → 逐格向下填充并 copy 字体/边框/对齐 → 另存临时 xlsx → 再用 pandas 重新读取**。这是全链路最昂贵的一步（openpyxl 全量读写 + 逐单元格 Python 循环）。

在生成一次综合看板过程中，**同一份 `xlsx_path` 被独立预处理/重载的次数约为 9 次**：

| # | 调用方 | 路径 |
|---|--------|------|
| 1 | `gen_compare_tool.compute_data` | `load_data` → `load_and_filter` → 预处理 |
| 2 | `gen_abs_cost_report.compute_data` | `read_data` → `load_and_filter` → 预处理 |
| 3 | `gen_spread_report.compute_data` | `read_data` → `load_and_filter` → 预处理 |
| 4 | `gen_institution_stats.compute_data` | 自有 `preprocess`（`shutil.copy2` + openpyxl 升级 + 再 preprocess）|
| 5 | `gen_investment_ledger`（2026） | `load_and_filter` → 预处理 |
| 6 | `fig6_credit_panel.render_credit_panel` | `preprocess_xlsx_for_pandas(ledger_path)` |
| 7 | `fig8_credit_total_panel.render_credit_total_panel` | `preprocess_xlsx_for_pandas(ledger_path)` |
| 8-9 | `fig7_wlz_panel`（regenerate=True） | 内部调 fig4_new + fig5，各自再 preprocess |

> 6MB 级综合看板（`ABS综合看板_20260715.html`）与近期文件体积暴涨，佐证数据量增大后该重复开销被线性放大。

### 影响
- 运行时间随台账规模线性叠加，**理论上约 8/9 的预处理时间是纯浪费**。
- openpyxl 逐格 `copy(font/border/alignment)` 是 Python 级循环，是 CPU 热点。

### 建议方向（不在本次改动）
1. **引入单次预处理 + 缓存**：在 `main()` 开头调用一次 `preprocess_xlsx_for_pandas(xlsx_path)`，得到"已展开合并单元格"的 DataFrame（或临时文件），通过参数注入各 `compute_data`，各子模块新增"接收已处理 df/tmp_path"的入口，跳过自身预处理。
2. **`load_and_filter` 增加 `preprocessed_df` / `tmp_path` 可选入参**：命中则跳过 openpyxl 阶段，直接走 pandas 清洗，向后兼容旧签名。
3. `preprocess_xlsx_for_pandas` 本身可优化：仅在"确有合并单元格"时才逐格填充；格式 copy 对 pandas 只读场景完全无用（pandas 不读格式），**可整体删除 font/border/alignment 的 copy**，仅保留 `.value` 填充，即可显著提速。

## 四、P2 — 稳健性与一致性隐患

### 4.1 临时文件清理不统一，存在泄漏风险
- `load_and_filter` 的调用方（compare/cost/spread/ledger）在 `finally` 里 `os.remove(tmp_path)`，做法正确。
- 但 `gen_institution_stats.preprocess` 走的是 `~/Documents/LikeCodeNex/tmp/{base}_25col.xlsx` 固定路径，**审计中未见对应清理**，会残留累积临时文件。
- `fig6/fig8/fig4/fig5` 的 `preprocess_xlsx_for_pandas` 返回的临时文件**是否每处都 remove 需逐一确认**（部分只 `wb.close()` 未删临时产物）。
- 建议：统一用 `tempfile` + 上下文管理器（`with`）或 `try/finally` 收口，杜绝路径分叉与泄漏。

### 4.2 异常处理粒度不一致
- `main()` 只捕获 `RuntimeError`（QC 未过），其余异常直接抛出。
- 但 `fig6` 用 `except Exception` 降级为占位块，`fig8` / `fig7` 却无兜底——**同为投资人分析子面板，容错策略不一致**，某一子面板异常会导致整份看板生成失败。建议对所有可选面板统一采用"失败降级为占位"策略。

### 4.3 局部 import 散落
`import re / import pandas as pd / import json` 出现在函数体内部（如 `build_integrated_html` 内 `import re`、`main` 内 `import pandas`）。虽不影响功能，但降低可读性，建议上提到模块顶部。

### 4.4 QC 每模块各跑一遍
每个 `compute_data` 内 `run_qc(..., precheck_only=True)` 基于各自重载的数据。改为共享数据源后，QC 可在单一数据集上集中执行一次，减少重复扫描。

## 七、综合看板 QC 体系设计（新增）

### 7.0 现状与盲区
现有 QC 分布在**各模块内部**（`gen_compare_tool.run_qc` / `gen_abs_cost_report.run_qc` / `gen_spread_report.run_qc` / `gen_institution_stats.run_qc`，统一基于 `abs_common.QCRunner` 的 ok/warn/fail 三级框架），编排层末尾另有一个"panel 数量 + JS 函数存在性"的轻量结构自检。

现有 QC 的**三大盲区**：
1. **跨模块一致性无人校验**：4 个模块各自独立读取同一台账，但"同一口径的数字是否互相对得上"（如优先A总认购份额、机构去重数、时间范围）从未被交叉比对。一旦某模块预处理/过滤逻辑漂移，只会静默产出不一致的看板。
2. **端到端产物校验过弱**：末尾仅数 panel 数量，未校验每个 panel 是否真的有数据、JS 数据是否可解析、悬浮球语料是否覆盖全年份。
3. **无输入闸门与无基线对比**：台账本身的结构/时效未在入口统一体检；与上一版看板的关键指标无回归对比，数据跳变无法被察觉。

建议在保持 `QCRunner` 框架不变的前提下，构建**四道 QC 关卡**（Gate），全部产出 ok/warn/fail 并汇总，fail 即中断生成。

---

### 7.1 Gate 0 — 输入体检（Pre-Ingest，`main()` 读数据前）
在单次预处理之后、分发给各模块之前，对台账做一次性入口体检：

| 检查项 | 级别 | 说明 |
|--------|------|------|
| 文件存在且可读、非 0 字节、后缀为 .xlsx | FAIL | 提前于 openpyxl 异常给出可读报错 |
| 列数 = 25 且列名与 `COLNAMES` 逐一匹配 | FAIL | 列错位是最隐蔽的数据污染源 |
| 关键列非全空（规模/簿记时间/分层情况/认购机构） | FAIL | 防止读到空表或错误 sheet |
| 合并单元格已全部展开 | FAIL | 校验预处理产物本身是否成功 |
| `簿记时间` 可解析比例 ≥ 阈值（如 95%） | WARN | 日期脏数据预警 |
| 台账最新簿记日期与"当前系统日期"跨度 | WARN | 提示台账是否为陈旧文件（防用错版本）|
| 优先A记录数 > 0 | FAIL | 无优先A则全链路无意义 |

> 收益：把"错文件 / 错列 / 空表"挡在 9 个模块之外，避免半路崩溃或产出错误看板。

---

### 7.2 Gate 1 — 模块内 QC（沿用现有，补强）
保留各模块 `run_qc(precheck_only=True)`，并统一补充以下通用断言（可下沉到 `abs_common` 复用）：

- **数值域校验**：成本 ∈ [1%, 10%]、利差 ≥ 0、认购份额 > 0、认购倍数合理区间。
- **黑名单剔除生效核验**：`EXCLUDED_INSTITUTIONS` 在全部 `EXCLUDED_ROLE_COLS` 中确无残留。
- **口径自洽**：`resolve` 后份额 ≥ 原始份额（WXY 穿透只增不减），分层过滤后无 优先A1/A2 误入。
- **NaN 预算**：各关键派生列的 NaN 占比不超阈值，超则 WARN。

---

### 7.3 Gate 2 — 跨模块一致性校验（**新增，最高价值**）
这是当前**完全缺失**的关卡。各模块 `compute_data` 完成后，编排层提取各自的"共享口径指标"做交叉比对：

| 一致性断言 | 参与模块 | 级别 |
|-----------|----------|------|
| **优先A总认购份额**（resolve口径）一致 | compare / cost / spread | FAIL（差异>0.001亿）|
| **优先A记录条数**一致 | compare / cost / spread | FAIL |
| **去重机构数**一致或差异可解释 | compare / institution | WARN |
| **簿记时间范围 [min,max]** 一致 | compare / cost / spread | WARN |
| **资产类型集合**一致 | compare / cost / spread | WARN |
| 2026 台账在 ledger 与 pricing 口径下的**项目数**一致 | ledger / compare | WARN |
| fig6/fig8 授信机构名经 `approximate_match` 后**无大面积未命中** | investor | WARN |

> 实现方式：各 `compute_data` 的返回 dict 统一附带一个 `qc_fingerprint`（含 total_share / rec_count / inst_count / date_range / asset_set 等标量），编排层集中比对。既轻量又能一举锁定"模块间口径漂移"。

---

### 7.4 Gate 3 — 端到端产物校验（写文件后，强化现有末尾自检）
在现有 panel 计数基础上补充：

| 检查项 | 级别 |
|--------|------|
| panel 数 = 期望数（10 + 年份数），且每个 (module, sub) 唯一无重复 | FAIL |
| 每个 panel **非空**（去标签后正文长度 > 阈值，非占位块） | FAIL/WARN |
| 每个模块的**子 Tab 按钮数 = 该模块 panel 数**（防按钮/面板错配） | FAIL |
| 内嵌 JSON（`data_js`/`proj_sizes_js` 等）可被 `json.loads` 解析 | FAIL |
| `selectModule`/`selectSub` 函数存在且默认激活 tab 有对应 panel | FAIL |
| 智能问答悬浮球语料**覆盖全部年份**（2024/2025/2026 均在语料内）| WARN |
| 输出文件大小落在合理区间（过小=渲染失败；暴增 N 倍=数据异常）| WARN |
| CSS 关键类名（各模块 CSS 均已拼入）无缺失 | WARN |

---

### 7.5 Gate 4 — 基线回归对比（可选增强）
与 `01_latest` 中上一版看板对比关键标量（优先A总份额、机构数、项目数、panel 数），跳变超过阈值（如 ±20%）即 WARN，并打印 diff。可有效捕捉"台账误替换 / 口径突变 / 某模块静默失效"。实现上可将每次生成的 `qc_fingerprint` 落一份 sidecar JSON（如 `ABS综合看板_YYYYMMDD.qc.json`），下次生成时读取对比。

---

### 7.6 QC 报告落盘
建议每次生成时输出一份结构化 QC 摘要（`*.qc.json` + 控制台汇总），沿用 `QCRunner.summary()` 的"PASSED / PASSED WITH WARNINGS / FAILED"三态，任一 Gate 出现 FAIL 即中断并非零退出。这样看板产物与其质检结论一一对应，可追溯、可回归。

## 八、下一步建议

建议按 P0 → P1 → P2 顺序推进：P0（单次预处理 + 注入）是运行时关键，需为 `load_and_filter` 及各 `fig*` 面板设计**向后兼容的"可注入预处理产物"参数**。

QC 体系可与 P0 改造**协同落地**——单次预处理产物天然成为 Gate 0/Gate 2 的统一数据源，各 `compute_data` 顺带产出 `qc_fingerprint`，边优化边加固，一次改造同时收获"更快 + 更稳"。改造后用同一份台账做前后运行时对比与四道 Gate 的回归验证。

**这套 Schema 的价值**：新问法不再是「加一条正则」，而是「往对应 slot 的 filler 里加一个模式或同义词」，且每个 filler 有独立单测，泛化边界清晰可控。

### 9.3 数据准确性：从「计算正确」升级为「计算正确 + 口径绑定 + 可溯源」

现有答案已「可复现」，但准确性还可再加三道保险：

**① 口径绑定（对接第七章 qc_fingerprint）**
问答用的 `window.ITL_ALL_DATA` 与看板卡片必须同源同口径。建议在 spec 计算后，把答案数字与看板对应筛选卡片做**运行时一致性抽查**（即最佳实践第 5 条「面板联动一致性」的自动化），任何 chat 数字都能在下方面板复现，否则标注「口径待核」。可复用第七章的 `qc_fingerprint` 作为「问答与看板同源」的校验锚点。

**② 答案可溯源**
每个数字答案附「数据出处」：命中记录数 N、聚合口径（份额 V 列合计 / 平均成本）、时间与年份范围。现有条件芯片已做了一半，补上「参与计算的记录数 + 口径一句话」即可让用户完整核对——这比数字本身更能建立信任。

**③ 置信度分级与兜底**
为解析结果打**置信度**：全部 slot 由规则精确命中 = 高；用了子串兜底/同义词/LLM 填槽 = 中；仅部分识别 = 低。
- 高：直接给数字。
- 中：给数字 + 显式提示「我的理解是：〔条件芯片〕，若不对请换个说法」。
- 低/未识别：走现有「没听懂 + 例句」，**绝不硬算**（继承最佳实践第 2 条）。

### 9.4 语义捕捉通用性的「兜底智能」：可选 LLM 填槽（离线可关）

若要突破正则天花板又不牺牲确定性，推荐**「规则优先、LLM 兜底填槽」**的旁路：

1. 规则先跑，`recognized=true` 且置信度高 → 直接用，**不调 LLM**（绝大多数问句走这条，零成本）。
2. 规则 `recognized=false` 或置信度低 → 才调 LLM，**且 prompt 严格约束**：只允许输出一个 JSON spec（字段白名单 = 现有 slot），每个值必须落在「本地词表/年份枚举/指标枚举」内，附一个 `confidence`。
3. LLM 返回后**逐字段校验**：机构名不在词表→丢弃该字段；年份不在 2024-2026→丢弃；指标不在枚举→拒答。校验通过的 slot 才进 `filterRecs`。
4. 全程**数字仍由本地台账算**，LLM 从不接触/输出数字。可通过开关完全禁用，退化为纯离线规则模式（默认离线场景保持关闭）。

这样 LLM 只当「翻译官」不当「计算器」：它把「理财子今年投得咋样」翻译成 `{category:理财子→机构集合, year:2026, metric:share}`，翻错了用户能从芯片看出来，翻对了扩大了覆盖度，且**任何情况下数字都是真的**。

### 9.5 回归护栏：把「准确性」变成不可回退的工程契约

现有 harness（[`itl_chat_harness.js`](../skills/ABS工具箱/scripts/lab/itl_chat_harness.js)）已是问答版自检，建议再强化为**准确性金标准回归**：

- 将文档第二节 19 题（含正确答案数字）固化为 `itl_chat_questions.txt` + 期望输出快照（P2-5）。
- 每题不只比「答没答」，还断言**具体数字**与**条件芯片内容**，任何偏差即 fail。
- 新增分层单测：每个 slot filler 独立喂 20~30 变体问法，度量识别率——这是「语义通用性」的量化指标，改动前后不许下降。
- 接入第七章 QC 的 Gate 3（端到端产物校验）：看板产出后自动跑一遍问答金标准题，chat 数字与看板卡片交叉比对，双向锁死口径。

### 9.6 落地优先级（与文档 P0/P1/P2 对齐，不重复造轮子）

| 阶段 | 动作 | 收益 |
|---|---|---|
| 立刻（继承文档 P0） | 修 B5/B6/H1/G1 四个「确定错数」——它们直接损害信任，且都是小改动 | 消除高危误答 |
| 近期（文档 P1 + 本章 9.3） | 裸年份范围/分别分组/极值方向/占比/阈值 + 答案可溯源(记录数+口径) + 置信度分级 | 覆盖度↑、可核对性↑ |
| 结构化（本章 9.2） | 正则重构为声明式意图 Schema + slot 单测 | 泛化边界可控、可维护 |
| 可选增强（本章 9.4） | LLM 兜底填槽旁路（默认关，带字段白名单校验） | 突破正则天花板，零幻觉风险 |
| 工程化（本章 9.5 + 文档 P2-5） | 数字级金标准回归 + slot 识别率度量 + 接入 Gate 3 | 准确性成为不可回退契约 |

**一句话结论**：不建议让 LLM 直接回答金融数字。最优机制是**「语义层可以聪明甚至用 LLM，计算层永远是确定的本地台账」的物理隔离**，配合口径绑定、可溯源、置信度分级三道保险，以及数字级金标准回归护栏——在最大化语义通用性的同时，把数据准确性锁成工程契约。

## 九、问答机器人 · 数据准确性与语义通用性的更优方案（新增）

> 针对 [`itl_chat.js`](../skills/ABS工具箱/scripts/itl_chat.js:288) 的 NL2Spec 纯规则解析器。现有设计已相当扎实（离线、可回归、宁可拒答不硬猜、条件芯片可核对），本章不推翻，而是提出**在保留其优点前提下的分层增强机制**。

### 9.0 现有架构的本质与边界

现状是一条**单遍正则流水线**：`parse()` 顺序抽取 年份→时间→指标→分组→极值→阈值→实体，各维度用独立正则各管一段，最后拼成 `spec`，再 `filterRecs + agg` 本地计算。

- **优点**：全离线、零依赖、毫秒级、答案 100% 可复现、有 harness 回归基座。这些是金融工具的生命线，任何新方案都**必须继承**。
- **本质瓶颈**：正则是「模式枚举」，每种新问法都要新增一条正则。文档第二节 19 题里 ❌/⚠️ 有 11 处，根因几乎都是**同一类**——某维度的表达变体没被枚举到（"去年"没拼时间、"分别"没触发分组、"最低"没识别方向、追问继承判据太粗）。这是规则系统的固有天花板：**覆盖度靠人力堆正则，无法泛化**。

### 9.1 核心判断：不引入 LLM 直连，改用「三层解析 + 可信数据绑定」

金融场景里，把自然语言直接交给 LLM 生成数字是**高风险**的（幻觉 = 自信的错误数字，正是本模块最忌讳的）。更优机制是**让语义层更聪明、让计算层永远确定**，两者物理隔离：

```
用户问句
  │
  ├─[L1归一化层] 同义词/别名/类别 → 标准值（可选 LLM 仅在此层，且只输出受控词表 token）
  │
  ├─[L2 意图槽位层] 抽成结构化 slots：{维度筛选, 时间, 指标, 分组, 对比, 极值, 阈值, 追问}
  │        └ 规则优先；规则未覆盖时，才调 LLM「填槽」——但 LLM 只能输出 spec 的合法字段，
  │          绝不输出任何数字或结论
  │
  └─[L3 确定计算层] 现有 filterRecs + agg 原样保留 —— 数字永远来自本地台账，可复现、可回归
```

**关键约束**：LLM（若引入）的输出被严格限制为「填 spec 槽位」，且每个槽位值必须能在词表/枚举里校验通过，否则丢弃回退到规则。这样即便语义层出错，也只会「筛错范围」（用户能从条件芯片一眼看出），**永远不会编造一个不存在的数字**。

### 9.2 语义通用性：从「枚举正则」升级为「意图 Schema + 可插槽解析器」

把散落的正则重构成**声明式意图 Schema**，每个维度是一个 slot filler，可独立测试、独立扩展：

| Slot | 现状（正则位置） | 通用性升级方向 |
|---|---|---|
| `entity` | `matchEntities()` 词表 + 子串兜底 | ✅ 已从数据自动生成，最泛化的一层，保留 |
| `synonym/alias` | 内联 `.replace('夹层','中间级')` | 抽成 `SYNONYM` 常量表（P2-4），业务同事可提词表 PR |
| `category` | 无（I3 误伤） | 新增 `INST_CATEGORY` 规则表：`理财子=/(理财)$/`、`券商自营` 等，命中→展开为机构 OR 集合 |
| `time/year` | `parseTime/parseYears` | 收敛为单一时间解析器，输出 `{years[], monthSpec}` 正交结构（根治 B5 年月冲突、B6 裸相对年） |
| `metric` | `parseMetric` 顺序判定| 计数意图前置（P0-4）；引入「意图明确度」打分而非硬顺序 |
| `groupBy/compare` | `parseGroupBy` | 「多值实体 + 分别/各自/谁多」→ 自动按该字段分组（P1-2） |
| `extreme` | `EXTREME_*_RE` | 已支持方向，补「哪家/谁+指标→自动分组取头名」（P1-3） |
| `ratio` | 无（E1 只给分子） | 新增占比 slot：分子=全条件，分母=去主语实体保留时间（P1-4） |
| `threshold` | 仅 share | 扩展到 cost/spread 阈值（P1-5） |
| `followup` | `hasEntityFilters` 判据 | 拆 entity/year 独立继承（P0-3），根治 H1 高危误答 |

**这套 Schema 的价值**：新问法不再是「加一条正则」，而是「往对应 slot 的 filler 里加一个模式或同义词」，且每个 filler 有独立单测，泛化边界清晰可控。
