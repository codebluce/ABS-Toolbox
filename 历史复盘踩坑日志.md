## 坑 #1：新增数据写入U/V列覆盖原值

### 现象
初期直接将簿记明细数据写入台账的U/V列（认购机构/认购份额），覆盖了原始认购数据。

### 根因
未区分"认购"和"申购"概念，将明细申购数据直接写入了台账的认购列。

### 修复
新数据只写入W列（申购利率）、X列（穿透机构）、Y列（申购规模），U/V列绝对不碰。

---

## 坑 #2：多利率拼接在同一单元格

### 现象
同一机构不同利率的申购被拼接到同一行，如 "1.65%/1.70%/1.75%"，无法区分。

### 根因
为了节省行数，将同机构所有利率合并到一个单元格。

### 修复
不同利率分行展示，同机构连续排列。

---

## 坑 #3：优先A/B交叉排列

### 现象
优先A和优先B的申购行交替排列，不符合台账格式规范。

### 根因
仅按机构分组，未区分分层（优先A/优先B）。

### 修复
优先A行全部在前，优先B行全部在后，不交叉。

---

## 坑 #4：openpyxl清空残留旧数据

### 现象
旧区域的行数多于新区域行数时，旧行末尾的数据未被覆盖，残留旧值。

### 根因
写入前未显式清空目标区域的全部单元格。

### 修复
写入前对目标区域每个格子显式设为 None。

---

## 坑 #5：多项目时中间保存导致数据丢失

### 现象
逐个项目处理并保存时，后一个项目覆盖前一个项目的修改。

### 根因
对同一个文件多次 openpyxl 加载/保存。

### 修复
全部在内存中计算完成后一次性写入，只做一次保存。

---

## 坑 #6：杭州联合银行等机构名未自动匹配

### 现象
明细中"杭州联合银行"与台账"杭州农商行（资管）"因名称不同而无法匹配。

### 根因
缺乏机构名映射规则。

### 修复
引入硬编码映射表 + 精确匹配优先策略。

---

## 坑 #7：明细中直接匹配的机构名走了hard_map

### 现象
明明明细机构名与台账完全一致（如"中信证券"），却先走了硬编码映射表改成了错误的名称。

### 根因
精确匹配优先级低于 hard_map。

### 修复
先精确匹配，再 hard_map，再模糊匹配。

---

## 坑 #8：字符串利率与数值利率混排序崩溃

### 现象
W列既有数值利率（如 0.0165）也有字符串（如"自持"），统一排序时报类型错误崩溃。

### 根因
Python 无法比较 float 和 str。

### 修复
分桶处理：数值按利率排序，字符串按机构名排序，合并时数值在前、字符串在后。

---

## 坑 #9：多项目拼接偏移导致数据错位/丢失/A列降序

### 现象
多项目批量处理时，A项目的WXY行侵入下游B项目的区域，B项目的部分原始行丢失，A列序号出现降序。

### 根因
拼接逻辑中 `min(adjusted_start, start_row)` 取最小值导致读取范围收缩。

### 修复
从前往后逐段拼接，中间行以原始行号索引读取，项目跳过用 `end_row+1`，不做偏移补偿。

---

## 坑 #10：新增数据从行1开始覆盖了表头

### 现象
写入新行时从行1开始，覆盖了台账表头。

### 根因
写入循环未预留表头的两行。

### 修复
数据从行2开始写入（`r = i + 2`）。

## 坑 #11：表头WXY列标题值为空

### 现象
WXY列的数据正确，但表头行（Row 1~2）的WXY列标题文字缺失。

### 根因
`orig_headers` 读取时WXY列值本来就是 None，写入新数据后未单独设置表头文字。

### 修复
在格式统一优化步骤中，显式设置 WXY 列的表头文字。

## 坑 #12：向下填充丢失单元格格式

### 现象
向下填充后，填充的单元格字体、对齐、边框与源行不一致。

### 根因
填充时仅复制了值（value），未复制源单元格的 Font/Alignment/Border。

### 修复
填充时同步复制源单元格的 Font/Alignment/Border。

## 坑 #13：新增行WXY列与前列格式不一致

### 现象
WXY列的字体、对齐方式与左侧A~V列不一致。

### 根因
WXY列是新增列，未继承台账已有列的格式。

### 修复
读取V列格式作为模板，应用到 WXY 列的 font/alignment/border。

---

## 坑 #14：WXY列和格式统一的字体/对齐冲突

### 现象
WXY列前处理设置了字体/对齐，最后一步格式统一优化又覆盖了一遍，导致两处设置冲突。

### 根因
WXY列前处理时做了格式设置，后续 `apply_unified_format` 又全表重设。

### 修复
WXY列前处理只设边框和数字格式，字体/对齐交给最后一步统一处理。

## 坑 #15：N列ALL-IN成本3位小数被覆盖成2位

### 现象
N列原始数据为三位小数（如 2.870%），格式统一后变成两位（如 2.87%）。

### 根因
N列与M/CD列共用 `0.00%` 格式。

### 修复
N列独立使用 `0.000%`，不与M列共用格式。

## 坑 #18：增量追加时多项目拼接偏移导致数据错位（2026-05-11）

### 现象
在已有WXY数据的v1文件基础上，用 `--multi` 继续录入新项目时，**A项目的WXY行侵占了B项目的原始区域**。

具体表现：
- 已录入禾昱9-2(白条)和盛澜1-3(京企贷)WXY数据的v1文件
- 追加录入盛赢3-6(企业主贷)和耘睿21-4(金条)
- 结果：盛赢3-6和禾昱9-2之间的区域混乱，禾昱9-2的未匹配WXY行写入了盛赢3-6区域，而盛赢3-6的原始U/V数据被推到下游

### 根因分析

**拼接待发逻辑缺陷**：`process_multi` 函数中的拼接逻辑使用了一个错误的 `min(adjusted_start, start_row)` 读取策略

```python
# 旧代码（有bug）
for start_row, end_row, project_name in project_ranges_forward:
    adjusted_start = start_row + offset
    max_read = min(adjusted_start, start_row)  # ← 这里取min，当项目扩展后adjusted_start > start_row，min取了start_row，反而少读了！
    if max_read > current_row:
        for r in range(current_row, max_read):
            final_rows.append(dict(orig_data[r]))
    ...
    current_row = adjusted_start + new_len  # ← current_row跳到偏移后的位置，导致中间行丢失
```

当A项目（禾昱9-2，靠前）因WXY行数扩展而变长（offset>0），靠后项目（盛赢3-6）的 `adjusted_start > start_row`。此时 `min(adjusted_start, start_row) = start_row`，**反而比不偏移时读得更少**，导致A项目的扩展WXY数据被当作了"中间行"写入，同时也少读了非项目中间行。

且当前项目区域的数据读取范围用 `for r in range(current_row, start_row)` 只读到原始起始行，但A项目扩展后的WXY尾行已经越过了 `start_row`，导致数据混乱。

### 修复方案

将拼接逻辑简化为**纯原始行号索引，不做偏移补偿**：

```python
# 新代码（正确）
for start_row, end_row, project_name in project_ranges_forward:
    # 追加 current_row 到 start_row-1 的原始行（原始行号索引）
    for r in range(current_row, start_row):
        if r <= max_orig_row:
            final_rows.append(dict(orig_data[r]))
    
    # 追加新构建的项目行（已在内存中处理好）
    new_rows = project_new_rows.get(project_name, [])
    final_rows.extend(new_rows)
    
    # 跳过项目的原始区域（无论新行数多少，都以原始end_row为基准）
    current_row = end_row + 1
    offset += new_len - orig_len  # 只用于追踪，不再参与行号计算
```

核心原则：
1. 中间行**始终用原始行号索引**读取（project_start_row之前的数据）
2. 插入项目新行后，current_row直接跳到原始项目区域的end_row+1
3. **不做偏移补偿**（即不用 `adjusted_start = start_row + offset` 作为读取依据）

### 验证方法
1. 从原始预处理文件一次性录入4个项目（禾昱9-2+盛赢3-6+盛澜1-3+耘睿21-4）
2. 检查每个项目边界：
   - A项目结尾 → B项目起始：seq字段不同、name不同、A的U/V/WXY正常结束、B的WXY从头开始
   - B项目结尾 → C项目起始：同上
   - 非项目区域（如云泓76号、栗泽21号等）：不被影响

### 预防措施
- `process_multi` 函数中的拼接逻辑禁止使用 `adjusted_start` 偏移量来计算读取范围
- 永远以 `start_row`（原始行号）为基准读取中间行，以 `end_row+1` 跳过已处理区域
- 新增项目时优先从预处理后的原始文件一次性处理所有项目，避免增量追加
- 必须增量追加时，确认输入文件中的WXY列没有侵占下游项目区域后再执行

## 坑 #19：手动处理尚赢2-4（多sheet格式）时WXY追加行破坏了原始T/U/V列（2026-05-11）

### 现象
尚赢2-4的簿记明细是多sheet格式（优先A/优先B分属不同sheet），脚本不支持，需手动处理。手动写入时出现了两个问题：

**问题1（已修复）：原始U列被重复使用**  
明细机构"交银理财"在优先A和优先B都有多笔申购，同机构"建信信托"也有多笔申购。手动匹配脚本对同一个原始U/V行（如"建信信托 V=0.05"）匹配了多笔明细（建信信托0.0172/0.0178/0.018/0.0183等多笔），导致原始U列被复制使用多次。

**问题2（已发现并修复）：T/U/V列被向下填充扩散到追加行**  
追加的未匹配明细行全部堆在原始区域末尾，但手动脚本在写入后执行了全表的A~T列向下填充，导致追加行（本应属于优先A/优先B分层）的T列从上一行继承成了受益权的T值（0.0515），U/V列也从原始行继承到了错误的机构名。

### 根因分析

**问题1根因**：匹配逻辑中，当明细中多行匹配同一个原始U（如明细有3笔"建信信托"，原始台账只有1行"建信信托"），脚本没有对匹配过的U/V做"一次使用"保护。

```python
# 错误行为
for item in items_a:
    if item in used: continue
    if match(ledger_org, item.org):  
        used.add(item)  # 只标记了明细项已用
        # 但原始U/V行（建信信托）在后续循环中又被同一个明细匹配了
```

但实际更严重的问题是——**没有对原始U行进行"一次使用"限制**。原始U/V行（如"建信信托 V=0.05"）只有一行原始数据，当多笔明细中都包含"建信信托"时，它们都去匹配这一原始U行，就导致了U/V重复。

**问题2根因**：手动脚本在写入新行后，对整个尚赢2-4区域执行了 `A~T列` 的向下填充。但追加行（未匹配明细）的A/B/C/E/F等列需要从原始行填充（继承项目名、序号、分层等），而T/U/V列本应跳过。脚本没有区分"哪些列需要填充、哪些列不能碰"。

### 修复方案

**针对问题1**：匹配逻辑增加原始U/V的"一次使用"保护——同一个原始U/V行只能匹配一条明细，一旦匹配成功，该原始行标记为已用，后续明细不能再匹配它。

**针对问题2**：追加行按分层分组排列（优先A追加跟在优先A段后、优先B追加跟在优先B段后），且向下填充只在**同分层内进行**，T/U/V列不参与填充：

```python
# 按分层处理
sections = {}  # 每个分层一个section
for layer, rd in orig_rows:
    sections.setdefault(layer, []).append(out)

# 追加未匹配明细到对应分层
remaining = [i for i in items_a if i not in used]
sections['优先A'].extend(remaining_a_rows)
sections['优先B'].extend(remaining_b_rows)

# 填充只在同分层内进行
for section in sections.values():
    for i in range(1, len(section)):
        for col in range(1, 21):  # 只填充A~T列（不含U/V）
            if section[i][col] is None:
                section[i][col] = section[i-1][col]
```

### 预防措施
1. **优先使用脚本**：对于标准3列格式的簿记明细，脚本都能处理。只有在遇到多sheet/多格式时才需要手动。
2. **手动处理前，先尝试将多sheet合并为标准3列格式**，然后走脚本流程。
3. **匹配原始U/V时，必须加"一次使用"保护**：一个原始U/V行只能被一条明细匹配。
4. **追加行必须按分层归位**：优先A追加行跟在优先A段后，不要一股脑堆在末尾。
5. **向下填充范围限定在A~T列**，U/V列不参与向下填充。
6. **输出后，用v2版比对验证T/U/V列未被篡改。**

## 坑 #20：手动处理后T/U/V列被篡改的验证方法（2026-05-11）

### 现象
对尚赢2-4手动录入WXY数据时，虽然WXY列数据正确，但原始T/U/V列被向下填充篡改：
- T列：追加行T值全变成了受益权的0.0515（应为优先A的0.0167/优先B的0.0183）
- U/V列：追加行继承了原始U/V值，导致同一机构出现多次

### 根因
手动处理时，向下填充的列范围过大（整个A~T列），且追加行未按分层归位。T/U/V列的值被原始行的值向下扩散污染。

### 验证方法
对比v2（无WXY的原始版本）和v3（有WXY的手动版本）中尚赢2-4区域的T/U/V列：
```python
# 对比原始版本和手动版本中同一项目的T/U/V列
v2: T=0.0167(优先A)/0.0183(优先B)/0.031(中间级)/0.0515(受益权)
v3: 应与v2完全一致，不得改变
```

### 修复验证
1. 确认尚赢2-4的T列值按分层正确：优先A=0.0167、优先B=0.0183、中间级=0.031、受益权=0.0515
2. 确认U列中每个机构不重复出现
3. 确认下游项目（耘睿21-1→耘睿24-7）的边界完整，无偏移

### 后续预防
1. 手动脚本在写入和填充时，必须限制填充范围，T/U/V列跳过
2. 输出后用对比脚本自动验证原始A~V列未被篡改
3. 优先将多sheet明细合并为标准格式后走脚本流程

## 坑 #23：明细机构用法定全称，台账用简称，匹配静默失败（2026-05-11）

### 现象
耘睿21-2的明细中，穿透机构全部使用法定全称（如"中信证券股份有限公司（投行上报）""杭州联合农村商业银行股份有限公司（资产管理部）"），台账中对应的是简称（"中信证券投行""杭州联合银行"）。现有匹配策略（精确匹配→hard_map→包含→去后缀）没有任何一条能处理此类全称。脚本没有报错，21笔优先A中仅匹配了2笔（有U行的招银理财/民生理财），其余全部静默追加到末尾。原始U行的WXY空着，看起来像"尚未录入"而非"匹配失败"。

### 根因
缺乏**全称→简称的自动截断**能力。

### 修复
在机构名映射规则中增加第5条"全称截断匹配"：
1. 去掉"股份有限公司""有限责任公司"等公司后缀词
2. 提取括号前的前两段核心名称（如"杭州联合农村商业银行"→去掉"农村商业银行"→"杭州联合"）
3. 再去与台账进行包含/去后缀匹配
4. 同时hard_map增加常见全称→简称对作为兜底

### 预防
- 脚本中实现全称截断匹配逻辑
- 处理含"股份有限公司""有限责任公司"等全称的明细时，先检查匹配率
- 发现匹配率异常低（如耘睿21-2的21行优先A只匹配了2行）时，应立即排查机构名匹配问题，不要跳过检查

## 坑 #24：多sheet+宽表组合场景，列结构与标准3列不同（2026-05-11）

### 现象

3. 簿记录入只处理增量项目，存量项目完全不动
4. QC 7.2验证存量项目WXY数据完整保留
